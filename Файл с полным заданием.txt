Индивидуальная разработка веб-приложения (full-stack)
1. Общая постановка задачи
Разработать полнофункциональное веб-приложение, реализующее бизнес- процессы выбранной предметной области. Приложение должно состоять из серверной части (обработка запросов, бизнес-логика, работа с данными) и клиентской части (интерфейс взаимодействия с пользователем).
Архитектура:
SPA (Single Page Application):
клиент на JavaScript/TypeScript c использованием фреймворка (React, Vue, Angular и т. д.) и отдельный серверный АРІ (REST, GraphQL и т. п.).
2. Общие функциональные требования (для всех вариантов)
2.1. Аутентификация и авторизация
Регистрация нового пользователя (логин, пароль, email). Вход в систему (логин/пароль).
Выход из системы.
Доступ K основным операциям только для аутентифицированных пользователей (публичные страницы - только вход/регистрация и, возможно, просмотр общедоступных данных, если это предусмотрено вариантом). 
2.2. Модель данных и бизнес-логика
Доменная модель должна содержать минимум 4 сущности. Обязательно использование связей «один-ко-многим» И «многие-ко-многим"
Бизнес-логика должна быть выделена B отдельные классы/модули(сервисы), а не располагаться в контроллерах или представлениях.
Для взаимодействия с пользователем (передача данных между сервером и клиентом) необходимо использовать DTO (Data Transfer Objects).
2.3. CRUD-операции
Реализованы операции создания, чтения (список и детали), обновления и удаления как минимум для двух основных сущностей (например, товары и заказы, посты и комментарии, книги и выдачу). Для связующих сущностей (таблиц связей) CRUD может быть частичным (например, только создание и удаление).
2.4. Фильтрация и поиск (на двух страницах)
Страница А список объектов, принадлежащих текущему пользователю (например, «Мои заказы», «Мои заметки», «Мои посты»). На этой странице реализованы: 
поиск по текстовому полю (название, описание и т.п.).
фильтрация по категории / тегу;
фильтрация по диапазону дат (если применимо) или по значению числового поля (например, цена, размер).
Страница В
На этой странице реализованы: публичный список объектов (всех пользователей или общедоступных). 
поиск по названию / заголовку ;
фильтрация по автору / владельцу (частичное совпадение имени или выбор ещё один дополнительный фильтр (по тегу, по цене, по дате - в зависимости варианта).
Фильтрация и поиск должны выполняться на стороне сервера (через параметры запроса), результат должен обновляться без полной перезагрузки страницы

На клиенте: все поля форм проверяются перед отправкой (обязательность, минимальная длина, корректный формат email, пароль не менее 6 символов и т.д.). Ha сервере: некорректные запросы отклоняются с кодом 400 и возвращают понятное сообщение об входящие данные обязательно ошибке. проходят валидацию,

2.7. Технические требования

Код должен быть структурирован, читаем, с осмысленными именами переменных и функций.
Данные должны сохраняться между запусками приложения (файловая база, SQLite, PostgreSQL, MySQL - на выбор студента). При использовании mock-сервисов обязательно наличие файла файловой базы с тестовыми данными.
Серверная часть должна реализовывать корректные HTTP-методы (GET, POST, PUT, PATCH, DELETE) и возвращать соответствующие коды ответа.

3. Критерии оценки
Оценка выставляется на основе выполнения требований раздела 2 и специфики выбранного варианта.
На оценку «удовлетворительно» (3)
Доменная модель содержит не менее 4 сущностей с использованием связей «один-ко-многим» и/или «многие-ко-многим». Приложение реализовано в архитектуре МѴС или SPA (целостное веб- приложение с серверной и клиентской частью).
Имеется не менее трёх функционально различных страниц. Реализованы CRUD-операции как минимум для двух основных сущностей
(создание, чтение списка, чтение деталей, обновление, удаление).
Используется язык JavaScript (допускается TypeScript) и система сборки (для ЅРА) или встроенный шаблонизатор (для МѴС). Присутствует адаптивная вёрстка. Серверная часть обеспечивает хранение данных (файл или база данных) и

Валидация реализована и на клиенте, и на сервере (сервер отклоняет некорректные данные с кодом 400). На оценку «отлично» (5)
Выполнены все требования на «хорошо».
Нет предупреждений при сборке запуске (линтер, консоль браузера, логи сервера).
Реализованы две страницы с фильтрацией и поиском B точномсоответствии с п.2.4 (все указанные виды фильтров работают корректно и удобны для пользователя).
Реализована полноценная аутентификация и авторизация (разграничение доступа).
Для всех операций (включая фильтрацию) предусмотрены индикаторы загрузки и понятные сообщения об ошибках.
Приложение корректно обрабатывает ошибки HTTP (400, 401, 403, 404, 500) и выводит пользователю информативные сообщения.
